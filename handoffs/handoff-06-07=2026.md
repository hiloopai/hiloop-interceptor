## What Exists Today

This repository is a strong Phase 0 contract and stdio interception walking skeleton. It is not
yet a general-purpose or production-ready interception system.

The current runtime path is:

```text
wrapped child process
  -> captured stdout/stderr
  -> RawSignal
  -> StdioLogNormalizer
  -> canonical hiloop_core::event::Event
  -> JsonlExporter

optional:
  RawSignal -> JsonlRawStore
```

The pipeline centrally stamps common provenance, including fork identity, wrapper identity,
normalizer identity, raw source/kind, retention policy, and available process metadata.

### `hiloop-core`

`hiloop-core` owns concepts intended to be shared across public and private components:

- typed `RunId`, `ForkNodeId`, `ForkOrdinal`, `ForkPath`, and `ForkContext`;
- `ChildOrdinalAllocator`;
- `Hlc` and `HlcClock`;
- the canonical logical `Event`;
- narrow event names, attribute keys, payload digests, and media types;
- deliberately narrow scalar `AttributeValue` variants.

It should own persisted and wire contracts. It should not become a general home for orchestration
or wrapper implementation traits.

### `hiloop-interceptor`

`hiloop-interceptor` currently provides:

- a CLI and child-process supervisor;
- fork-context resolution and child environment injection;
- pass-through mode;
- bounded Tokio stdio capture and normalization pipeline;
- generic stdio normalization;
- JSONL event export;
- optional JSONL raw observation retention;
- wrapper-local `Source`, `Normalizer`, `Exporter`, and `RawStore` seams;
- shared provenance-key constants instead of scattered magic strings.

### Current E2E Coverage

The new compiled-binary mock-harness E2E suite covers:

- pass-through context injection;
- stdout/stderr preservation and child exit-code propagation;
- capture and raw-observation linking;
- lossless per-stream ordering under load;
- non-UTF-8 and empty-line handling;
- LF, CRLF, and final partial-line handling;
- exporter/raw-store flushing before nonzero child exit is returned;
- failure before child start for invalid raw-output configuration;
- failure before child start for conflicting output paths.

The fixture is a deterministic POSIX shell harness:

## Settled Design Decisions

Do not reopen these without a concrete reason or new evidence:

1. `hiloop-core` owns shared persisted/wire concepts, not orchestration.
2. Wrapper-only traits stay in `hiloop-interceptor`.
3. Private-only orchestration traits stay near their implementations in the private monorepo.
4. The canonical `Event` is a logical/searchable model, not a ClickHouse table shape.
5. Storage connectors may use efficient physical formats and explicit projections.
6. Types should be as narrow as practical.
7. Use `Option<T>` for absence rather than sentinel empty strings.
8. Raw ingress may remain looser than normalized contracts until its taxonomy stabilizes.
9. Normalization is generic-first, with optional harness-specific semantic enrichment later.
10. The default pipeline uses bounded Tokio queues and lossless blocking behavior.
11. JSONL is a local development and testing exporter, not the intended production backend.
12. Cargo is supported for local development in this repository.
13. The private polyglot monorepo should be Bazel-first using Bazel 9.x, Bzlmod, and `rules_rust`.
14. The private monorepo should pin the exact interceptor git commit and use `hiloop-core` from the
    same commit.
15. Performance should initially be recorded, not used as a PR gate.
16. Use the conventional Rust dual license: `MIT OR Apache-2.0`.

## Open Design Questions

These are ordered roughly by how much they constrain the next implementation work.

### 1. Telemetry Failure Modes

Define explicit required and best-effort modes.

Questions:

- When may telemetry failure affect the wrapped process exit result?
- In best-effort mode, how is loss made visible rather than silently ignored?
- In required mode, when may the wrapper terminate or block the child?
- How are source, normalization, raw-store, and exporter failures distinguished?

Recommended direction: support explicit required and best-effort policies; never allow silent loss.

### 2. Raw Observation Contract

Current stdio "raw" storage is not an exact source replay contract:

- it loses original newline framing;
- it is stored after normalization policy is evaluated;
- parser/normalizer failures can prevent raw preservation;
- raw observation IDs are only locally convenient identifiers.

Define:

- exact source-native bytes and framing;
- globally unique and stable raw observation identity;
- source identity and source-local sequence;
- observed and ingested timestamps;
- process and fork context;
- the point at which an observation is considered durable.

Recommended direction: when preservation is enabled, store source-native observations before
normalization.

### 3. Event Identity and Delivery Semantics

The current `Event` lacks the identity needed for durable retry and deduplication:

- stable event ID;
- stable raw observation ID;
- source-local sequence;
- ingest timestamp;
- explicit retry/partial-delivery semantics.

Decide these before production export.

### 4. Canonical Event Scope

Decide how much source-specific data belongs in the canonical event.

Recommended direction: make `Event` a searchable common projection and retain source-native raw
observations for lossless replay. Do not attempt to make one universal schema losslessly represent
every future protocol.

### 5. Normalization and Enrichment Split

The current router can run every supported `Normalizer`. This allows generic normalization and
early semantic extraction to coexist, but it mixes two responsibilities.

Before adding a second real semantic format, split the stages into:

```text
SourceNormalizer -> canonical base events -> SemanticEnricher(s)
```

An enricher failure should not destroy valid base events or retained raw observations.

### 6. Production Export Boundary

Decide whether the wrapper writes directly to storage or to a versioned ingest service.

Recommended direction: export to an ingest gateway or collector. Keep database credentials,
physical schema, retries, and deduplication in the backend rather than in every customer-side
wrapper.

### 7. Supervisor Lifecycle

Production supervisor behavior is not defined or tested for:

- signal forwarding;
- process groups;
- descendant cleanup;
- orphan reaping as PID 1;
- parent disappearance;
- wrapper cancellation;
- precedence between telemetry failure and child exit.

Harden this before treating the wrapper as a reliable PID-1 supervisor.

### 8. Batching and Live Latency

Current partial batches export only when the batch fills or the input closes.

Add:

- maximum count;
- maximum bytes;
- maximum delay;
- explicit memory-bound accounting;
- capture-to-export latency metrics and tests.

### 9. Raw Privacy and Redaction

Raw observations can contain secrets and customer data. Define retention, encryption, redaction,
and access controls before proxy or broad raw capture lands.

### 10. Extension Stability

Decide whether `Source`, `Normalizer`, `Exporter`, and `RawStore` are internal implementation seams
or supported third-party plugin APIs.

Recommended direction: keep them unstable/internal through the first production vertical slice.

### 11. Existing Environment Merge

The wrapper currently overwrites `OTEL_RESOURCE_ATTRIBUTES`. Define typed merge and conflict
policies for existing OTEL, proxy, and CA-bundle environment variables.

### 12. Next Vertical Slice

Recommended sequence:

1. Harden process supervision.
2. Define raw observation, event identity, and telemetry failure contracts.
3. Add an OTLP receiver.
4. Add a mock versioned ingest gateway and end-to-end delivery tests.
5. Add proxy capture only after privacy and raw-retention contracts are settled.

### Additional Known Gaps

- Normalization diagnostics are counted and then discarded.
- `Source` and `Backpressure` exist but the stdio runtime does not use them directly.
- JSONL raw IDs such as `raw-jsonl-1` are not globally stable.
- JSONL `flush()` is not a durable `fsync()` guarantee.
- OTLP ingest, proxy capture, ClickHouse delivery, eBPF, and production replay do not exist yet.

## Recommended Immediate Goal

Do not start with ClickHouse or a Codex-specific normalizer. First define and test the contracts
that make all later capture and delivery trustworthy:

```text
telemetry failure policy
  + exact raw observation identity/durability
  + stable event identity/deduplication
  + supervisor lifecycle behavior
```

Once those are explicit, the next practical vertical slice is:

```text
mock harness
  -> hardened interceptor
  -> embedded OTLP receiver
  -> mock versioned ingest gateway
  -> query/assert captured fork-stamped events
```
