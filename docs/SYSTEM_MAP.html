Phase 0 system map

A trustworthy telemetry wrapper for branchable agent runs.

The repository currently proves a complete stdio walking skeleton: supervise a child, capture observations, normalize them into fork-stamped events, optionally retain raw observations, and flush JSONL output. The interfaces are designed for more sources and exporters, but several product-level contracts must be decided before those extensions become stable.

Current state: contract-rich, source-light

Strong identity, event, pipeline, and test foundations. Only stdio capture and local JSONL delivery are implemented. OTLP, proxy, production ingest, replay, and full PID-1 supervision remain future work.

Repository boundaries

Who owns what

The architecture deliberately separates durable shared contracts, wrapper-local implementation seams, and private backend orchestration.

hiloop-core

Stable shared vocabulary consumed across repositories.

  • Fork identity and canonical paths
  • Hybrid logical timestamps
  • Logical event schema
  • Future wire/protobuf contracts

hiloop-interceptor

Open-source edge runtime and its extension boundaries.

  • CLI and child supervision
  • Capture sources and normalization
  • Raw retention and delivery
  • Local mocks, JSONL, and E2E tests

Private monorepo

Product orchestration and backend implementation.

  • Control plane and fork lifecycle
  • Ingest gateway and persistent stores
  • Sandbox runtime and snapshots
  • Branch-diff and product APIs
Interface catalog

Contracts and extension points

These are the principal types and traits shaping the system. Some are durable shared contracts; others are intentionally wrapper-local and may still evolve.

ForkContext

hiloop-core / shared contract

The branch-observability spine stamped onto children and every canonical event.

  • RunId identifies one experiment tree
  • ForkNodeId identifies one branch node
  • ForkPath supports ancestor/prefix queries
struct ForkContext {
  run_id: RunId,
  fork_node_id: ForkNodeId,
  fork_path: ForkPath,
}

Event

hiloop-core / logical contract

The backend-independent searchable projection of one normalized observation.

  • Narrow signal and attribute types
  • Always fork-stamped
  • Optional out-of-row payload reference
struct Event {
  ts: Hlc,
  fork identity,
  signal: SignalType,
  name: EventName,
  attributes: Attributes,
  payload_ref: Option<PayloadRef>,
}

Source

hiloop-interceptor / capture boundary

Produces ordered raw signals from a capture mechanism.

  • Implemented as a trait
  • Not yet used by stdio runtime capture
  • Needs cancellation and backpressure semantics
trait Source {
  fn name(&self) -> &'static str;
  fn signals(&self) -> RawSignalStream;
}

RawSignal

hiloop-interceptor / loose ingress

A deliberately flexible envelope before semantic normalization.

  • String source/kind taxonomy
  • Raw bytes and source-local attributes
  • Current stdio version loses newline framing
struct RawSignal {
  source: String,
  kind: String,
  observed_at: Hlc,
  attributes: Map<String, String>,
  body: Bytes,
}

Normalizer

hiloop-interceptor / semantic extraction

Declares support and turns one raw signal into zero or more canonical events.

  • Versioned descriptor and schema output
  • Can request raw preservation
  • Currently conflates source normalization and enrichment
trait Normalizer {
  fn descriptor(&self) -> Descriptor;
  fn supports(&self, raw: &RawSignal) -> Support;
  async fn normalize(...) -> Outcome;
}

NormalizationOutcome

hiloop-interceptor / result contract

Returns derived events, diagnostics, and a raw-retention request.

  • Diagnostics are currently counted, then discarded
  • Retention is aggregated across normalizers
  • Needs failure-isolation rules for enrichers
struct NormalizationOutcome {
  events: Vec<Event>,
  diagnostics: Vec<Diagnostic>,
  raw_retention: RawRetentionPolicy,
}

RawStore

hiloop-interceptor / bronze retention

Persists replayable source observations and returns a reference for derived events.

  • JSONL implementation exists
  • Currently stores only after normalization succeeds
  • Durability and globally unique IDs are unresolved
trait RawStore {
  async fn store(context, raw)
    -> RawObservationRef;
  async fn flush();
}

Exporter

hiloop-interceptor / delivery boundary

Delivers already-normalized batches and flushes before clean shutdown.

  • JSONL and in-memory implementations exist
  • Partial failure and retry identity are unresolved
  • Recommended production target: ingest gateway
trait Exporter {
  async fn export(&self, events: &[Event]);
  async fn flush(&self);
}

SemanticEnricher

future wrapper-local extension

Optional harness-aware extraction layered after generic source normalization.

  • Should never be required for useful base telemetry
  • Must not destroy generic events on failure
  • Supports customer-specific harness formats
trait SemanticEnricher {
  fn supports(raw, events) -> Support;
  async fn enrich(raw, events)
    -> Outcome;
}
Current implementation

One captured run, step by step

This sequence reflects the path exercised by the compiled-binary E2E suite when --events-jsonl is enabled.

1 Resolve configuration

CLI parses fork identity, output paths, and child argv. Output-file conflicts fail before spawn.

Fail before side effects
2 Prepare child

Supervisor injects HILOOP_* and replaces OTEL_RESOURCE_ATTRIBUTES with fork identity.

Environment merge policy open
3 Spawn and pipe

Child stdout/stderr are piped; output is simultaneously written back to parent stdout/stderr.

Harness bytes preserved
4 Frame observations

Each stream becomes line-shaped RawSignals. CR/LF delimiters are removed; very long lines split at 64 KiB.

Raw replay semantics open
5 Normalize and stamp

StdioLogNormalizer creates log events; the pipeline centrally stamps process, wrapper, normalizer, raw-source, and fork provenance.

Central provenance enforcement
6 Optionally preserve raw

With --raw-jsonl, the pipeline stores each RawSignal and attaches raw.observation_id to derived events.

Currently after successful normalization
7 Batch and export

Events are written when a batch fills or input closes. Partial batches remain buffered during a long-running quiet child.

Maximum live latency open
8 Drain and return

The wrapper waits for the child, capture tasks, and pipeline; flushes; then returns the child exit code unless telemetry failed.

Signal/failure semantics open

Traits ahead of runtime

Source and Backpressure exist but are not used by stdio capture. Normalizer routing supports multiple implementations before stage semantics are settled.

Raw is not fully raw

Stdio framing strips newline bytes before retention and raw storage happens after normalization. This is insufficient for exact replay and parser-failure debugging.

Production supervision

No signal forwarding, process-group lifecycle, orphan reaping, descendant cleanup, or complete PID-1 behavior.

Product-shaped telemetry path

No OTLP receiver, proxy, lifecycle events, ingest gateway, durable production exporter, branch-diff query, or privacy/redaction policy.

Terms

Working glossary

Raw observation

Source-native captured information used for replay, parser evolution, and debugging. The exact fidelity and durability promise is still an open design decision.

Canonical event

A fork-stamped logical telemetry projection designed for search, comparison, routing, and storage conversion. It should not be confused with a physical ClickHouse or OTLP shape.

Source normalizer versus semantic enricher

A source normalizer understands a capture protocol such as stdio or OTLP. A semantic enricher optionally understands a harness-specific format and adds higher-level meaning.

Required versus best-effort telemetry

Required telemetry makes completeness part of run correctness. Best-effort telemetry prioritizes harness continuity but must make loss explicit and measurable.
